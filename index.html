One of my large projects as an in-house developer for a consulting firm is a managed survey data dashboard. Our consultants prepare data and automated reports for our clients, who access them through our website. We are a small (7 people) office in a niche industry, so I work closely with all of the consultants in improving and updated the administrative portion of the website.

A big goal of ours was to create a streamlined product that we could process with a standard workflow, even when the underlying data varied considerably. One of the tools I developed fairly early on was a scripting tool that our consultants could use to write macros that perform common or repetitive data transformations. The macros would run automatically when new data was imported from a datafile or API, and the transformations would be essentially self-documenting.

The Problem

It was Ruby. Not Ruby-like or Ruby-inspired - plain old, eval to a proc on the server Ruby. This is dangerous for numerous obvious reasons, but I was pretty confident that no one in our company would want to maliciously attack the server, and moreover, they wouldn't know how if they did want to.

However, we're starting to license the system to other consulting firms who will act as service providers, so the need to harden this very very soft spot is fairly urgent. Additionally, our consultants have expressed a desire to better understand what the scripts our that I tell them to type in so they can write their own complicated logic, but teaching them the ins and outs of a general purpose programming language is not currently on my list of priorities. They're not developers, but they have a reasonably good understanding of SPSS Syntax and Excel formulae, so I figured I see what I could give them in that vein.

Having recently read Tom Stuart's Understanding Computation, I felt I could make a reasonable stab at developing a simple expression language in a short amount of time, the requirements being:

  1. Single-line expressions only (but nested functions are permitted)
  2. Must compile to a Ruby proc that takes a hash-like object as an argument (in our specific case, a Mongoid document)
  3. Evaluation only (the compiled proc would not be allowed to mutate any data - I am handling this within the Rails app)
  4. Simple enough syntax and a small enough API to learn in 15 - 30 minutes for a non-developer
  5. Can do everything we have already done in the last 12 months using the Ruby macros.

I used Treetop to write a grammar specification and implemented the language behind it. I'll go into a bit more detail later on, but first, let's briefly look at the syntax:

A basic list/array: [1 2 3]. I purposefully decided the value separator should be a single space rather than a comma or some other character, simply because it's one fewer thing for non-programmers to remember.

Here are some syntax examples along with the Ruby code they replaced.

The basic lookup function: remember, this operates on hash-like objects that implement the square brackets operator (most our data is stored by an id reference to a MySQL table, hence the integer keys).

Old style: r[276], r[:external_id] New style: {276}, {external_id}

A series of variables can be looked up as a list like this: {101 102 103 external_id}

A reference object lookup: In the case of our dashboard, each respondent (or 'case' or 'row' depending on what data analysis world you live in) belongs to a dataset, which can provide some default variables or perform calculations or analyses over the entire cohort.

Old style, calculating age: r.survey.date.year - r[276] New style: {{year}} - {276}

Functions: there is a very basic set of functions carefully chosen so we could easily re-implement all of our Ruby macros in Sorabji. The syntax is simply a keyword followed by a list (of either [...] or {...} form).

Finding the first present value in a list of possible values, the 'default' function:

Old style: r[101] || r[102] || r[103] New style: default{101 102 103}

You can mix lookups with integer or other values using the square bracket list:

default[{101} 2001].

This functions similar to Ruby's hash#fetch ({}.fetch(:key, default_value)).

Conditionals are ternary functions: Another nod to the Excel crowd:

Old style: r[101] < 100 ? 1900 + r[270] : r[270] New style: if[({101} < 100) (1900 + {270}) {270}]

Any sort of reduce or inject function is much simplified over its Ruby equivalent:

[101, 102, 103].inject(0) {|sum, x| sum + r[x].to_i }
# try explaining why that needs to be so complicated
sum{101 102 103}
mean[{101} 10 20]

Implementation

For the most part, I followed Tom Stuart's example of creating an Abstract Syntax Tree using Ruby structs. Each class would implement a to_proc function that determined how it should evaluate the hash being fed into it. Most classes are very simple. Each abstract class was paired with a corresponding Node class that inherited from Treetop::Runtime::SyntaxNode. I might eventually factor those out, but I found it useful in developing the language to have one set of classes concerned with how the tree is parsed and one concerned with compiling to Ruby procs.

Here is the Integer Literal class:

class IntegerLiteralNode < ASTNode
  def to_ast
    IntegerLiteral.new(text_value.to_i)
  end
end

class IntegerLiteral < Struct.new(:value)
  def to_proc
    ->(*args){ value }
  end
end

Obviously, the functional representation of an integer value should only ever return the integer value, so the proc is exceedingly simple. The corresponding grammar:

rule integer
  [0-9]+ <Sorabji::IntegerLiteralNode>
end

Yay! I should point out that I have never done this before, and went to school for an entirely unrelated subject, so excuse me while I find it incredibly cool that I can feed in a string like '9' to a parser and have it evaluate to a Ruby integer without using the built-in eval.

Similarly simple primitives like booleans, strings, operators, comparators, etc. have very similar implementations. I'm not going to go through every grammar rule step-by-step here, but the entire specification is available in the repo.

Fast forward a dozen or so commits. I decided that most of my grammar rules fall into one of three categories: entities (anything at all, really), value entities (entities which, when evaluated as a proc, return a single value), and operations.

Value Entities include:

-bracketed expressions -reference object lookups ({{year}}) -object lookups ({276}) -integers, strings, and booleans -functions

It took numerous revisions to eventually get to this schema, but once I arrived, I found it was pretty easily extensible. The really interesting part turned out to be mathematical operations, which, due to our weird obsession with infix notation, is WAY less intuitive than I expected.
